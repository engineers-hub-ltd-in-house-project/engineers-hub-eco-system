# engram x TechFolio 統合: 市場調査・競合分析 指示書

## 目的

engram（エンジニア性格診断）と TechFolio（エンジニアリソース管理）の統合が、SES/フリーランスエンジニア市場において実需を持つかを検証する。本調査の結果をもって、統合開発への Go/NoGo 判断を行う。

## 調査期間

4週間（Week 1-4）

---

## 調査構成

```
Week 1        Week 2         Week 3           Week 4
─────────────────────────────────────────────────────
[競合・類似事例分析]         [分析整理]
              [SES企業ヒアリング 5-10社]
              [エンジニアアンケート 50名+]
                                              [レポート作成・Go/NoGo判断]
```

## Part A: 競合・類似事例分析（Week 1-2）

### A-1. 国内競合のパーソナリティ機能調査

以下の各サービスについて、下記の観点で調査する。

**調査対象**:
- Crowd Agent
- Matchbox
- Findy
- offers（過去のサービス含む）
- LAPRAS
- Forkwell
- レバテック

**調査観点**:

| 項目 | 確認内容 |
|------|---------|
| パーソナリティ診断機能の有無 | 診断の有無、次元数、質問数 |
| マッチングへの利用方法 | スキルのみか、性格特性も加味しているか |
| チーム構成分析機能 | プロジェクトチームの相性分析があるか |
| エンジニア公開プロフィール | ポートフォリオに性格特性を表示する機能があるか |
| AI面談支援機能 | 面談代行、自動応答等の機能があるか |
| 料金体系 | 企業向け課金モデル（月額/成果報酬/従量） |

**成果物**: 競合比較マトリクス（上記項目 x 各サービス）

### A-2. 海外類似事例の調査

以下の統合事例について、導入効果と課題を調査する。

| 事例 | 調査ポイント |
|------|------------|
| Pymetrics + LinkedIn | ゲーム型診断とプロフェッショナルプロフィールの統合方法、導入企業の評価 |
| Plum + ATS連携 | パーソナリティ診断と採用管理システムの統合アーキテクチャ |
| Predictive Index | 行動特性診断のチームビルディングへの活用方法 |
| Crystal Knows | パーソナリティ予測のコミュニケーション最適化への応用 |
| TestGorilla | スキル + パーソナリティの複合テストプラットフォーム |

**各事例の調査項目**:
- 診断の科学的根拠（心理測定理論、妥当性研究の有無）
- 企業への導入実績と公開されている効果データ
- エンジニア/候補者側の受容度（NPS、利用率等の公開情報）
- プライバシー対応の方針（データ開示粒度、オプトイン設計）
- 収益モデル

**成果物**: 海外事例サマリ（事例ごとに1ページ以内）

### A-3. 学術的根拠の調査

ソフトウェア開発チームにおけるパーソナリティとパフォーマンスの関係に関する先行研究を調査する。

**検索キーワード**:
- "personality traits software development team performance"
- "Big Five software engineer team composition"
- "psychometric assessment IT project success"
- "personality-job fit programmer"

**確認すべき知見**:
- どのパーソナリティ次元がチームパフォーマンスと最も相関するか
- パーソナリティの多様性とチーム成果の関係（多様性仮説 vs 類似性仮説）
- 自己報告式診断の限界と対策（社会的望ましさバイアス等）
- 測定結果の時間的安定性（再テスト信頼性）

**成果物**: 関連論文リスト（10-20本）+ 主要知見の要約

## Part B: SES企業ヒアリング（Week 2-3）

### B-1. 対象企業の選定基準

- SES事業、フリーランスエージェント、ITコンサルティング企業
- 所属/管理エンジニア数: 30名以上（統計的に意味のあるチーム構成が存在する規模）
- 目標: 5-10社

### B-2. ヒアリング項目

所要時間 30-45分を想定。以下の4セクションで構成する。

#### セクション1: 現状のエンジニア選定プロセス（10分）

1. 案件にエンジニアをアサインする際、どのような情報を参照しているか
   - スキルシート / 面談記録 / 過去案件実績 / その他
2. スキル以外の要素（コミュニケーション力、ストレス耐性、チーム適性等）をどの程度考慮しているか
   - 全く考慮しない / 面談で主観的に判断 / 何らかの指標がある
3. ソフトスキルの評価に使っているツールや手法があれば教えてほしい

#### セクション2: プロジェクトミスマッチの実態（10分）

4. 直近1年間で、アサイン後にミスマッチ（早期離脱、パフォーマンス不足）が発生した頻度
   - 全案件の何%程度か
5. ミスマッチの主な原因は何か（複数回答可）
   - スキル不足 / コミュニケーション問題 / 作業スタイルの不一致 / ストレス耐性 / その他
6. ミスマッチ1件あたりの概算コスト（再アサインの工数、クライアントへの影響等）

#### セクション3: パーソナリティデータへの関心（10分）

7. エンジニアの性格特性（10次元: 技術志向性、問題解決スタイル、コミュニケーション特性、チーム役割傾向、ストレス対応パターン、学習スタイル、仕事スタイル、リーダーシップ志向、曖昧さ耐性、デバッグ力）のデータがあった場合、アサイン判断に使いたいか
   - 5段階: 全く使わない - 非常に使いたい
8. 上記10次元のうち、案件マッチングに特に重要だと思う次元はどれか（3つまで選択）
9. チームの構成バランス（次元のカバレッジ、多様性）を可視化する機能があれば使いたいか

#### セクション4: 導入意向と価格感度（10分）

10. 既存のエンジニア管理ツールは何を使っているか
    - Excel/スプレッドシート / 自社システム / 外部SaaS（具体名） / なし
11. スキルマッチ + パーソナリティマッチを組み合わせたツールに月額いくらまで支払えるか
    - 無料のみ / ~1万円 / ~3万円 / ~5万円 / ~10万円 / 10万円以上
    - 管理エンジニア数に応じた従量課金の場合、1人あたりいくらが上限か
12. 導入にあたっての懸念事項は何か（複数回答可）
    - エンジニアの受容性 / データの信頼性 / プライバシー / コスト / 既存ワークフローとの統合 / その他

### B-3. 記録と整理

- 各ヒアリングは議事録を作成し、上記12項目の回答を構造化して記録
- 定量項目（5段階評価、選択式）は集計表にまとめる
- 定性コメントは項目別に分類・要約する

**成果物**: ヒアリング集計レポート

## Part C: エンジニアアンケート（Week 2-3）

### C-1. 対象と配布方法

- 対象: SES/フリーランスで稼働中のエンジニア
- 目標回答数: 50名以上
- 配布チャネル: 自社ネットワーク、SNS、エンジニアコミュニティ
- 回答時間: 5分以内

### C-2. アンケート設問

#### 属性情報（4問）

| # | 設問 | 回答形式 |
|---|------|---------|
| Q1 | 現在の働き方 | 選択: SES正社員 / フリーランス / 派遣 / その他 |
| Q2 | エンジニア歴 | 選択: 1-3年 / 3-5年 / 5-10年 / 10年以上 |
| Q3 | 主な技術領域 | 複数選択: Web Frontend / Web Backend / Mobile / Infra/SRE / Data/ML / その他 |
| Q4 | 直近1年の案件数 | 選択: 1件 / 2-3件 / 4件以上 |

#### パーソナリティデータの開示意向（5問）

| # | 設問 | 回答形式 |
|---|------|---------|
| Q5 | 性格診断（働き方の傾向やチーム適性を測る100問のテスト）を受けてみたいか | 5段階: 全く思わない - 非常に思う |
| Q6 | 診断結果を案件マッチングに活用されることに抵抗はあるか | 5段階: 非常に抵抗がある - 全く抵抗がない |
| Q7 | 診断結果を企業に開示する場合、どの粒度が許容できるか | 選択: 全データ（10次元パーセンタイル） / ハッシュタグ（8個のキーワード）のみ / 案件との相性ランク（高/中/低）のみ / 一切開示したくない |
| Q8 | 公開ポートフォリオ（技術経歴書の公開ページ）に診断結果を表示したいか | 5段階: 全く思わない - 非常に思う |
| Q9 | 診断を受ける動機として最も強いものはどれか | 選択: 自分の強み/弱みの理解 / 案件マッチング精度の向上 / ポートフォリオの充実 / チームでの役割理解 / 特にない |

#### マッチング機能への期待（3問）

| # | 設問 | 回答形式 |
|---|------|---------|
| Q10 | 案件を選ぶ際、スキル適合以外にどの情報がほしいか | 複数選択: チームの雰囲気 / 働き方のスタイル / プロジェクトの進め方 / メンバーの性格傾向 / 特にない |
| Q11 | 過去にアサイン後のミスマッチ（スキル以外の理由）を経験したことがあるか | 選択: ある / ない |
| Q12 | Q11で「ある」の場合、原因は何だったか | 複数選択: コミュニケーションの不一致 / 作業スタイルの違い / 期待役割のずれ / チーム文化の不適合 / その他（自由記述） |

### C-3. 分析方針

- Q5-Q9: 開示意向の分布を確認し、エンジニア歴・働き方との交差分析
- Q6 x Q7: 抵抗感と許容粒度の関係
- Q10-Q12: ミスマッチ経験者と未経験者での回答傾向の差異
- 自由記述: テキストマイニングによるキーワード抽出

**成果物**: アンケート集計レポート + クロス分析表

## Part D: 統合分析とGo/NoGo判断（Week 4）

### D-1. 分析フレームワーク

以下の3軸で統合判断を行う。

#### 軸1: 市場性（需要の存在）

| 判断基準 | Go条件 |
|---------|--------|
| SES企業のパーソナリティデータ活用意向（Q7平均） | 3.5/5.0 以上 |
| エンジニアの診断受診意向（Q5平均） | 3.0/5.0 以上 |
| エンジニアのデータ開示許容率（Q7で「一切開示したくない」以外） | 60% 以上 |
| SES企業の支払い意思額中央値 | 月額3万円以上 |

#### 軸2: 競合優位性

| 判断基準 | Go条件 |
|---------|--------|
| 国内競合にパーソナリティ x チーム分析の組合せ機能が存在しない | 該当 |
| 海外事例で統合型プラットフォームの効果実証がある | 1事例以上 |
| 学術的にパーソナリティとチームパフォーマンスの相関が支持されている | 主要研究で正の相関 |

#### 軸3: 実現性

| 判断基準 | Go条件 |
|---------|--------|
| 技術統合の障壁（Service Binding、共通認証基盤）が低い | 両システムがCF Workers + Lucia Auth |
| engramの診断データの統計的妥当性に重大な欠陥がない | 先行研究との整合性あり |
| 倫理・法的リスクへの対処方針が明確 | オプトイン設計で対処可能 |

### D-2. 判断基準

- **Go**: 3軸すべてで条件を満たす
- **Conditional Go**: 1軸のみ条件未達 → 条件付きでPoC実施、未達軸を重点的に検証
- **NoGo**: 2軸以上で条件未達 → 統合を見送り、各プロダクト単独で価値最大化を図る

### D-3. 最終成果物

1. **調査レポート**（以下を含む）
   - 競合比較マトリクス
   - 海外事例サマリ
   - 学術的根拠の要約
   - SES企業ヒアリング集計
   - エンジニアアンケート集計・クロス分析
2. **Go/NoGo判断書**
   - 3軸の評価結果
   - 判断理由
   - Go の場合: Phase 1（PoC）への移行計画
   - NoGo の場合: 各プロダクト単独での代替戦略

## 補足: 調査実施上の注意事項

- ヒアリング対象企業には NDA を締結し、engram/TechFolio の具体的な実装詳細は開示しない
- アンケートでは個人を特定できる情報を収集しない
- 競合分析は公開情報の範囲で行い、リバースエンジニアリング等は行わない
- 海外事例の調査は英語の公式ドキュメント・プレスリリース・論文を一次情報源とする
